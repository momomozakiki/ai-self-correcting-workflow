### [decision] Documentation Standard — lazy per-doc folding + Episodic `CHANGELOG.md`
- **What:** Codified a *lazy* convention so docs stop growing unbounded. A doc stays a
  flat `docs/<name>.md` until it either splits (Rule of One Question / token budget) or
  its in-file Revision History passes ~8 rows; then it **folds** into `docs/<name>/`
  where `index.md` keeps the canonical frontmatter + only the latest ≤3 history rows,
  and the full history moves to a sibling `CHANGELOG.md`. That changelog is Episodic —
  `applies_when: "Never load…"` **and** machine-readable `exclude_from_ai: true` — so it
  stays outside the active token budget. Changes: GUIDE.md §6.3/§6.4 (renamed §6.4 to
  "Folding a document into a folder"; → v4.3); Progressive Disclosure Guide §4 taxonomy
  row + §9 + §10 tree, and its sample `load_md_files()` now `continue`s on
  `exclude_from_ai` before scoring (→ v4.1); new `templates/docs/CHANGELOG_TEMPLATE.md`;
  fold-procedure guidance in `templates/docs/DOC_TEMPLATE.md`; SKILL.md Phase-2 trigger +
  folding note; ROADMAP linter item extended (flag >8-row histories; skip `exclude_from_ai`).
- **Why:** The genuine unbounded-growth vector was the in-file Revision History table —
  it grows forever and triple-books with `git log` and this ledger. Relocating it to a
  never-loaded Episodic sibling keeps active docs lean without inventing a fourth
  changelog mechanism; encoding it in GUIDE §6 + the templates makes downstream adopters
  inherit the behaviour automatically. `exclude_from_ai` (not just prose `applies_when`)
  is what makes "never load" machine-enforceable, since the retrieval script never parses
  `applies_when`.
- **Refs:** GUIDE.md §6.3–§6.4 · `docs/Progressive Disclosure Documentation Guide.md`
  · `templates/docs/CHANGELOG_TEMPLATE.md` · plan `should-we-create-folder-rustling-deer`
