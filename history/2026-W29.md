### [doc] Documentation Standard — `last_validated` = last content review, not last edit
- **What:** Resolved a contradiction: `DOC_TEMPLATE.md` defined `last_validated` as
  "date you last confirmed the content is correct" while `GUIDE.md` §6.3/§6.5, the
  embedded template comment, and the `SKILL.md` mirror all said "refresh on every
  non-trivial edit." Reworded so a mechanical/frontmatter-only edit bumps `version` +
  adds a Revision History row but **leaves `last_validated` unchanged**; the field only
  refreshes on a genuine content review. Updated all six spots per the CLAUDE.md
  anti-drift rule: `GUIDE.md` §6.3 prose + §6.5 bullet + embedded-template comment
  (→ v4.5, `last_validated` 2026-07-18, Revision History row added),
  `templates/docs/DOC_TEMPLATE.md` comment, and `skills/adaptive-workflow/SKILL.md`
  prose + frontmatter-example comment.
- **Why:** The field was documented to mean two different things; the "every edit"
  reading lets a mechanical frontmatter backfill overstate how recently content was
  validated — a provenance-honesty bug. The fix is portable and inherited by all
  adopters.
- **Scope note:** `SKILL.md` and `DOC_TEMPLATE.md` got **no** doc-standard version
  bump — they live under `source_directories` (`skills`, `templates`), not
  `documentation_directories: ["docs"]`, and carry no `version`/`last_validated`/
  Revision History of their own. Only `GUIDE.md` (which opted into the standard) is
  version-bumped; the source-file edits are traced by this ledger entry. (This
  corrected a reviewer amendment that assumed `SKILL.md` was a governed doc.)
- **Refs:** `GUIDE.md` §6.3/§6.5 (v4.5) · `templates/docs/DOC_TEMPLATE.md` ·

### [config] Added `permissions.allow` allowlist to stop plan-mode approval prompts
- **What:** Neither `.claude/settings.json` nor the adopter template
  `templates/settings.json.hooks` had a `permissions` block — only `hooks`. With no
  allowlist, Claude Code prompts for approval on **every** `Bash` command, including
  read-only Phase-0/planning commands (`git status/log/fetch`, `python -m unittest`),
  because plan mode blocks `Edit`/`Write` but does not auto-approve `Bash`. Added a
  `permissions.allow` list of the workflow's read-only commands to both files, and
  documented the mechanism in a new `docs/claude-code-hook-integration.md` §5.1.1
  (→ v1.9, `last_validated` 2026-07-18, Revision History row added).
- **Why:** The missing allowlist is a config gap, not a Claude bug — the assumption
  that "plan mode should allow any command" is unsupported (a `Bash` call can still
  write files, so the harness must ask). An explicit read-only allowlist is the safe,
  portable fix and is inherited by adopters via the template. Deliberately avoided
  `bypassPermissions` (it suppresses prompts in every mode, defeating plan-mode safety).
- **Refs:** `.claude/settings.json` · `templates/settings.json.hooks` ·
  `docs/claude-code-hook-integration.md` §5.1.1 (v1.9) · plan
  `please-check-what-is-jaunty-waffle` · `python -m unittest discover -s tests`
